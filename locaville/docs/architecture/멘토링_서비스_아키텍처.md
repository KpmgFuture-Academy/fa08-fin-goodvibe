## 1. 시스템 컨텍스트

![서비스 아키텍처 전체](../images/architecture_context.png)

### 한눈 요약

| 영역 | 구성 | 비고 |
|---|---|---|
| **User** | 농업인 / 마을 관리자(이장) | 두 페르소나, 화면이 별도 |
| **Frontend** | Vercel + Next.js + React | `app_user` (농민용), `web_user` (이장용) 두 앱 |
| **Backend** | FastAPI + Python + Render | Routers → Services → Repositories 3계층 |
| **Data Platform** | Supabase | Object Storage(사진·영수증) / Relational DB(사용자·일지·증빙) / **pgvector(RAG, 운영중)** |
| **External API** | KAMIS / 기상청 / Kakao Local / OpenAI | 농수산물 가격, 날씨, 주소 변환, LLM/STT/TTS/Vision |

### 핵심 디자인 결정 (3가지)

1. **RDB 가 진실, AI 는 보조** — Todo·일지·증빙 같은 사실 데이터는 RDB. AI 는 입력 정제 / 답변 생성 / 안내 / 사진 코칭 역할에 한정. 직불금 컴플라이언스 도메인이라 AI 환각이 RDB 진실을 흔들면 안 됨.
2. **Storage / DBMS 중립화** — `STORAGE_MODE` (rdb/json) 와 `DB_SOURCE` (mysql/postgres) 두 변수로 의미 분리. repository 코드는 표준 SQL 만 사용해 양쪽 호환.
3. **벡터 DB 단일화 완료** — 옛 Chroma (로컬) → **Supabase pgvector 이전 완료** (2026-06-04). Relational + Vector 한 곳에서 운영하여 백업/스케일 관리 단순. 옛 Chroma DB 는 rollback 안전망으로 2주 보존.

## 2. 라이브 카메라 코칭 (촬영 전 실시간 가이드) — **신규 2026-06 / 최우선 시연**

증빙 사진의 품질은 직불금 통과율과 직결. 농민이 "어떻게 찍어야 하는지" 모르는 경우가 많아, 카메라가 켜진 순간부터 0.8초 간격으로 backend 가 사진을 평가해 가이드 문구를 실시간으로 띄우고 — 충분히 좋아지면 자동 셔터를 내림. 흔들림 감지 시 cancel.

`PHOTO_CRITERIA` × `JOB_EVIDENCE_TO_CRITERIA` 가 (job_cd, evidence_type) 별로 시행지침 9p 기준을 정의 (예: "배수물꼬 2주", "2~5cm 4회", "납품된 바이오차").

```mermaid
sequenceDiagram
    actor 농민
    participant FE as app_user<br/>(PhotoLiveCoachOverlay)
    participant BE as Backend<br/>(/photo/live-coach)
    participant OAI as OpenAI Vision
    participant DM as DeviceMotion<br/>(iOS)

    농민->>FE: 활동 선택 → 카메라 켜기
    FE->>FE: getUserMedia (HTTPS only)
    Note over FE: 라이브 미리보기 시작
    FE->>DM: requestPermission() (iOS 13+)
    DM-->>FE: granted | denied

    loop 0.8s 주기 polling
        FE->>FE: canvas capture (1 frame)
        FE->>BE: POST /photo/live-coach<br/>multipart {frame, job_cd, evidence_type, session_id}
        Note over BE: photo_guard_service<br/>_build_pic_criteria_block<br/>(PHOTO_CRITERIA[criteria_key])
        BE->>OAI: vision prompt<br/>(기준 + "쉬운 일상 말" 규칙)
        OAI-->>BE: {score, guidance_text, ready: bool, issues: [...]}
        BE-->>FE: 평가 결과
        alt ready (점수 ≥ 임계, 흔들림 없음)
            FE->>FE: 자동 셔터 (3-2-1 카운트)
            FE-->>농민: 촬영 완료 → §3 업로드 흐름
        else 흔들림 감지 (DM acceleration > 임계)
            FE->>FE: 카운트 cancel + "흔들리지 않게" 표시
        else 가이드 필요
            FE-->>농민: "조금 더 위에서 찍어주세요" 등
        end
    end
```

### 주요 의사결정

- **0.8초 polling** (SSE/WebSocket 아님): Render free plan + 농민 LTE 환경에서 안정성 우선. 1초 미만 응답 latency 가 vision API 비용 trade-off 와 균형.
- **자동 셔터 + 흔들림 cancel**: 농민이 "지금이 좋은 타이밍" 결정을 안 해도 됨. 흔들리면 자동 취소 → 흐릿한 사진 방지.
- **"쉬운 일상 말" prompt rule**: 시행지침은 전문 용어 ("배수물꼬", "2~5cm 4회", "경운"). 코칭 문구는 LLM 에 "어르신께 쉽게" 변환 강제 + 4가지 변환 예시 주입.
- **HTTPS-only 제약**: `getUserMedia` 는 브라우저 보안상 HTTPS / localhost 만. LAN 시연 시 Vercel 배포 + HTTPS URL 필수 ([known-limitations §1](../dev/known-limitations.md)).
- **세션 in-memory**: `session_id` 별 polling 상태가 backend in-memory dict. 다중 instance / 재시작 시 손실 — 시연용 OK, 정식 출시 시 Redis ([known-limitations §9](../dev/known-limitations.md)).

## 3. 증빙 업로드 + AI 라벨링 (라이브 코칭 후속)

사진을 찍으면 (라이브 코칭이 자동으로 셔터를 내리는 시점) OpenAI vision 이 "영수증인가 작업 사진인가" 분류 + `evidence_type` 별 기준 (`PHOTO_CRITERIA`) 합치 여부 + Pillow 가 GPS·시각·농민명 워터마크. 직불금 이행 점검 통과의 핵심.

> 사진 촬영 **전** 흐름 (라이브 코칭) 은 §2. 이 장은 셔터 이후 흐름.

```mermaid
sequenceDiagram
    actor 농민
    participant FE as app_user<br/>(PhotoInputScreen)
    participant BE as Backend<br/>(/evidence/upload)
    participant Pillow as Pillow<br/>(image_quality + 워터마크)
    participant OAI_VIS as OpenAI Vision
    participant OS as Supabase<br/>Object Storage
    participant DB as Supabase RDB

    농민->>FE: 사진 촬영 + 활동/evidence_type 선택
    FE->>BE: POST /evidence/upload<br/>multipart {file, farmer_id, work_date, job_cd, evidence_type}
    BE->>Pillow: 품질 검사 (블러/밝기/해상도)
    Pillow-->>BE: 합격/불합격 + 메시지
    alt 품질 불합격
        BE-->>FE: 422 + "다시 촬영해 주세요"
    else 합격
        BE->>OAI_VIS: image → 분류 prompt<br/>(receipt | photo) + evidence_type 기준 합치
        OAI_VIS-->>BE: {kind, evid_cd 후보, label, criteria_pass}
        BE->>Pillow: GPS+시각+농민명 워터마크
        Pillow-->>BE: 원본 + 워터마크 두 파일
        BE->>OS: PUT uploads/evidence/{key}
        Note over BE: identity resolve +<br/>parent journal 매칭 (없으면 자동 생성)
        BE->>DB: INSERT evidence<br/>(+ journal/prj_journal 옵션)
        DB-->>BE: PK (user_no, seq_no, job_date, exec_no)
        BE-->>FE: 201 + {evidence_id, ai_label, status}
    end
```

### 주요 의사결정

- **품질 검사 → vision → 워터마크 순서 고정**: 품질 불합격은 vision API 비용 낭비 없이 즉시 reject.
- **`raw_json` 컬럼에 status 보관**: 이장님 검토 결과(`pending`/`confirmed`/`rejected`) 가 evidence 테이블 스키마 변경 없이 들어감. 운영 변화 흡수 쉬움.
- **`PHOTO_CRITERIA` × `JOB_EVIDENCE_TO_CRITERIA` dict** (`photo_guard_service.py`): (job_cd, evidence_type) 조합별 시행지침 9p 기준이 hard-code. 표 데이터 RAG 추출 정확도 문제 회피 ([known-limitations §4](../dev/known-limitations.md)).

## 4. 영농일지 작성 (음성 → AI 정제 → DB)

또 다른 핵심 입력 흐름. 농민이 손에 흙 묻은 채 한 손으로 음성 녹음하면 LLM 이 정제해서 일지가 됨. 사업 작업과 자동 매칭되면 사업 일지로도 동시 저장.

```mermaid
sequenceDiagram
    actor 농민
    participant FE as app_user<br/>(VoiceInputScreen)
    participant OAI_STT as OpenAI STT
    participant BE as Backend<br/>(/ai/diary/refine, /diary)
    participant OAI_LLM as OpenAI GPT
    participant DB as Supabase RDB

    농민->>FE: 음성 녹음
    FE->>OAI_STT: audio (m4a/webm)
    OAI_STT-->>FE: transcript (str)
    FE->>BE: POST /ai/diary/refine<br/>{text, farmer_id, work_date}
    BE->>OAI_LLM: prompt + transcript
    OAI_LLM-->>BE: structured JSON<br/>{job_cd, parcel_no, exec_desc, ...}
    BE-->>FE: refined draft
    농민->>FE: 검토 후 저장
    FE->>BE: POST /diary<br/>{farmer_id, work_date, job_cd, parcel_no, exec_desc, ...}
    Note over BE: identity resolve<br/>(farmer_id → user_no, amo_regno)
    Note over BE: prj_todo_list 매칭<br/>(amo_regno + job_cd + 미완료 todo)
    alt 매칭 todo 있음
        BE->>DB: INSERT journal + prj_journal
    else 매칭 없음
        BE->>DB: INSERT journal
    end
    DB-->>BE: PK (user_no, job_date, exec_no)
    BE-->>FE: 201 Created<br/>{diary_id, prj_linked: bool}
```

### 주요 의사결정

- **STT 는 frontend → OpenAI 직접** (backend 거치지 않음): 음성 파일이 backend 를 거치면 latency 두 배. 사용자 경험 우선.
- **사업 자동 매칭**: 동일 `amo_regno` + `job_cd` + 미완료 todo 가 있으면 `prj_journal` 동시 생성. 농민이 "이게 사업 작업인지" 고민할 필요 없음.

## 5. 이장 대시보드 — 오늘 먼저 챙길 일 (laggard top 3) — **신규 2026-06**

옛 마을주민 대시보드 (§11) 가 "전체 현황 통계" 였다면, 새 대시보드는 **"오늘 누구를 챙길지 알려주는 순서표"**. laggard top 3 가 메인 주인공, KPI 4개는 보조. AI 가 "오늘의 운영 메모" 한 줄 생성 + TTS 로 들어보기.

```mermaid
sequenceDiagram
    actor 이장
    participant FE as web_user<br/>(/dashboard)
    participant BE_LAG as Backend<br/>(/admin/laggard-farmers)
    participant BE_TODO as Backend<br/>(/admin/todo-status)
    participant BE_AI as Backend<br/>(/ai/recommendation)
    participant BE_HELP as Backend<br/>(/farm-helpers)
    participant BE_EVID as Backend<br/>(/admin/recent-evidence)
    participant BE_TTS as Backend<br/>(/ai/tts)
    participant DB as Supabase RDB
    participant OAI as OpenAI

    이장->>FE: 대시보드 진입
    par 5개 병렬 호출
        FE->>BE_TODO: GET /admin/todo-status
        BE_TODO->>DB: ...
        DB-->>BE_TODO: todo items
        BE_TODO-->>FE: {items}
    and
        FE->>BE_AI: GET /ai/recommendation
        BE_AI->>OAI: GPT prompt (마을 요약)
        OAI-->>BE_AI: "박정호님 영수증 마감이 임박..."
        BE_AI-->>FE: {recommendation}
    and
        FE->>BE_EVID: GET /admin/recent-evidence?limit=8
        BE_EVID->>DB: ...
        BE_EVID-->>FE: gallery
    and
        FE->>BE_LAG: GET /admin/laggard-farmers?days=7&limit=8
        BE_LAG->>DB: SELECT 미이행 농가 + sample_todos
        BE_LAG-->>FE: laggards
    and
        FE->>BE_HELP: GET /farm-helpers
        BE_HELP-->>FE: helper pairs
    end

    Note over FE: priorityList 계산<br/>(top 3, 알림 보낸 농가 뒤로)
    FE-->>이장: 카운트 라인 + 우선순위 3 + KPI 4 + 챙길 농가 + 운영 메모

    alt 이장이 알림 발송
        이장->>FE: "문자로 알려주기"
        FE->>BE_LAG: POST /admin/laggard-farmers/{farmer_id}/notify
        BE_LAG->>DB: INSERT notification (recipient farmer)
        BE_LAG-->>FE: 200
        FE-->>이장: ✓ 알림 보냄 chip + 카드 뒤로 이동
    end

    alt 이장이 메모 듣기
        이장->>FE: "들어보기"
        FE->>BE_TTS: POST /ai/tts {text=recommendation}
        BE_TTS->>OAI: TTS API
        OAI-->>BE_TTS: audio blob
        BE_TTS-->>FE: audio/mp3
        FE-->>이장: 재생
    end
```

### 주요 의사결정

- **5개 병렬 fetch**: `Promise.race` 로 화면을 빨리 띄우고 나머지는 backgrounded. 농민 LTE 환경에서 체감 latency ↓.
- **laggard sort 룰**: 알림 보낸 농가는 뒤로 — "처리한 농가가 화면 위에 남지 않음" UX. 마감일 빠른 순 > 미이행 건수 많은 순.
- **운영 메모 안전망**: AI 추천 텍스트가 hero 농가/활동과 무관하면 fallback 한 줄 ("오늘은 마감이 가까운 활동부터…"). hero/메모 불일치 방지.
- **TTS fallback to Web Speech API**: OpenAI TTS 실패 시 브라우저 `speechSynthesis` 로 자동 대체 — 셀룰러 끊김 환경 대응.

## 6. RAG Q&A (질문 → intent 라우팅 → pgvector retrieve → 답변)

농민이 정책 질문하면 시행 문서(HWPX) chunk 를 retrieve 해서 농민 친근 어조로 답변. 질문 의도 자동 분류로 정확도 +α. **2026-06-04 부터 Supabase pgvector 운영**.

```mermaid
sequenceDiagram
    actor 농민
    participant FE as app_user<br/>(채팅 화면)
    participant BE as Backend<br/>(/ai/chat)
    participant OAI_EMB as OpenAI<br/>embeddings<br/>(text-embedding-3-large, dim=1536)
    participant VEC as Supabase pgvector<br/>(rag_chunks, HNSW cosine)
    participant OAI_LLM as OpenAI GPT

    농민->>FE: "저탄소 프로그램 참여하면 얼마 받아?"
    FE->>BE: POST /ai/chat<br/>{question, farmer_id?}
    Note over BE: intent 감지<br/>- money intent<br/>- date calc intent<br/>- 일반

    alt date calc intent
        Note over BE: → /ai/policy/calc 자동 라우팅<br/>(§7)
    else money / 일반 intent
        Note over BE: 검색어 보강<br/>(money intent → "지원금 단가 ha 원/ha" 추가)
        BE->>OAI_EMB: 검색어 → 1536-dim vector
        OAI_EMB-->>BE: vector
        BE->>VEC: SELECT ... ORDER BY embedding <=> $1 LIMIT 30
        VEC-->>BE: 30 chunks + cosine distance
        Note over BE: heading_path boost<br/>(activity +250, money +250)<br/>+ MMR rerank top-8
        BE->>OAI_LLM: system(농민 톤) + context(8 chunks) + question
        OAI_LLM-->>BE: 답변 텍스트
        Note over BE: 후처리<br/>- 인용 마커 제거<br/>- 천원→만원 변환<br/>- ellipsis 안전망
    end

    BE-->>FE: {answer, source_type, used_context[8]}
    FE-->>농민: 친근한 답변 표시
```

### 주요 의사결정

- **Intent 자동 분류 → 라우팅**: 날짜 계산 / 금액 / 일반 으로 분리해 각각 다른 prompt + retrieval 전략. 정확도 큰 차이 (예: 단가 질문 시 정답률 0 → 거의 100%).
- **dense embedding 외 keyword boost 병용**: 약어/고유명사("AWD", "바이오차") 가 dense 만으로는 약함. heading_path 토큰 매칭으로 강화.
- **Chroma → pgvector 이전 (2026-06-04)**:
  - `text-embedding-3-small` (1536) → **`text-embedding-3-large` + `dimensions=1536` 매개변수 native reduction** (HNSW 호환).
  - HNSW (cosine) index — 218 chunk 기준 평균 응답 385 ms.
  - `RAG_USE_PGVECTOR=0` env 로 옛 Chroma fallback (rollback 안전망, 2주 보존).
  - 10개 baseline 정확도 80% (5 명확 + 3 부분 + 2 부정확) — 시연 핵심 시나리오 모두 top-3 통과 ([known-limitations §4-A](../dev/known-limitations.md)).

## 7. 날짜 계산 자동 라우팅 (`/ai/policy/calc`)

농민이 "5월 27일 모내기 했어. 중간 물떼기 언제?" 처럼 구체 날짜 + 활동을 물으면 자동 감지해서 정밀 계산 endpoint 로 라우팅. 일반 RAG 답변보다 정확하고 단계별 추적 가능.

```mermaid
sequenceDiagram
    actor 농민
    participant FE as app_user
    participant BE as Backend<br/>(/ai/chat)
    participant CALC as calculate_policy_date<br/>(내부 함수)
    participant VEC as Supabase pgvector
    participant OAI as OpenAI GPT

    농민->>FE: "5월 27일 모내기 했어.<br/>중간 물떼기 언제부터 해?"
    FE->>BE: POST /ai/chat<br/>{question}

    Note over BE: _detect_chat_date_calc_intent
    Note over BE: ✓ "5월 27일" 매치<br/>✓ 참조 활동 "모내기"<br/>✓ 대상 활동 "중간 물떼기"<br/>→ date calc intent 확정

    BE->>CALC: calculate_policy_date(<br/>  question,<br/>  activity="중간 물떼기",<br/>  farmer_context={"모내기 날짜": "2026-05-27"})

    CALC->>VEC: retrieve k=12 + 활동 boost
    VEC-->>CALC: chunks (정책 룰 다양성 ↑)
    CALC->>OAI: calculation_prompt<br/>(3단계: 1.근거 인용  2.산정 과정  3.최종 답변)
    OAI-->>CALC: 3단계 텍스트

    Note over CALC: _light_clean_answer<br/>(sentence cutoff 없음 —<br/>다단계 답변 보존)

    CALC-->>BE: AIPolicyCalcResponse<br/>{answer (3단계), used_context}

    Note over BE: _extract_final_calc_answer<br/>("3. 최종 답변" 부분만)
    Note over BE: _strip_position_markers<br/>_strip_evidence_markers

    alt calc 성공
        BE-->>FE: {answer, source_type: "policy_calc"}
    else calc 실패 (RAG miss 등)
        BE->>BE: 일반 chat_with_rag 흐름으로 폴백 (§6)
        BE-->>FE: {answer, source_type: "rag"}
    end

    FE-->>농민: "모내기 +30일 = 6월 26일부터<br/>14~30일 진행, 7월 10~7월 26일 마무리"
```

### 주요 의사결정

- **chat 내부 fallback 패턴**: calc 가 retrieve 실패하거나 LLM 응답이 비정상이면 일반 chat 으로 자동 폴백.
- **3단계 답변 → 최종 답변만 노출**: 디버깅 가치를 위해 3단계로 생성, 농민에겐 3번만 보임.

## 8. Todo 조회 + 상태 계산 (computed_status)

농민 홈 화면의 "오늘 할 일". RDB 의 todo + 일지/증빙 합산으로 진척 상태를 derived. **AI 가 절대 끼지 않는 곳** — 직불금 영향이라 진실은 RDB 만.

```mermaid
sequenceDiagram
    actor 농민
    participant FE as app_user<br/>(HomeScreen)
    participant BE as Backend<br/>(/todo/today)
    participant DB as Supabase RDB

    농민->>FE: 화면 진입
    FE->>BE: GET /todo/today?farmer_id=1110000002
    Note over BE: identity_rdb.resolve_amo_regno<br/>(login_id / user_no / amo_regno 어느 형태든)
    BE->>DB: SELECT prj_todo_list<br/>LEFT JOIN ville_group, amo_family, user_master<br/>LEFT JOIN project, prj_activity, farm_job<br/>LEFT JOIN code_detail (job_progress_name)<br/>WHERE amo_regno=?
    DB-->>BE: todo rows (6건)
    BE->>DB: SELECT journal WHERE amo_regno=?
    DB-->>BE: 일지 (3건)
    BE->>DB: SELECT evidence WHERE amo_regno=?
    DB-->>BE: 증빙 (2건)

    Note over BE: todo + 일지 + 증빙 매칭<br/>→ computed_status 계산
    Note over BE: pending : 일지 X<br/>in_progress : 일지 있고 job_cmpl_yn=N<br/>completed : 일지 있고 Y, 증빙 충족<br/>missing_evidence : 일지 있고 Y, 증빙 부족

    BE-->>FE: {items: [<br/>  {todo_id, job_name, due_date,<br/>   computed_status, missing_evidence_types}<br/>]}
    FE-->>농민: 6개 todo 상태별 색상 표시
```

### 주요 의사결정

- **view 의존 제거 + raw JOIN (Phase C)**: 이전엔 `vw_jeotan_todo_board` view 호출. view DDL 동기화 부담 + DBMS 호환 이슈로 raw JOIN 으로 전환.
- **computed_status 는 backend 에서 derived**: RDB 컬럼으로 저장하지 않음. 일지/증빙 변경 시 자동 반영, 정합성 깨질 일 없음.

## 9. 농사 도와주기 (Helper ↔ Recipient 연결) — **신규 2026-06**

마을에는 디지털 기록이 어려운 어르신 농가가 있음. 같은 마을의 도와줄 농가가 어르신 대신 일지/증빙을 입력할 수 있도록 양쪽 동의 기반으로 연결. 입력된 일지/증빙은 **recipient** 의 user_no 로 저장 — 도우미는 매개체일 뿐.

```mermaid
sequenceDiagram
    actor 도우미 as 도우미 농가
    actor 어르신 as Recipient (어르신)
    actor 이장
    participant FE_H as app_user (helper)
    participant FE_R as app_user (recipient)
    participant FE_C as web_user (chief)
    participant BE as Backend<br/>(/farm-helpers/*)
    participant DB as Supabase RDB

    rect rgb(245, 240, 230)
        Note over 도우미,DB: 연결 요청 단계
        도우미->>FE_H: "도와드리기" 진입 + 대상 농가 검색
        FE_H->>BE: POST /farm-helpers/request<br/>{helper_user_no, recipient_user_no}
        BE->>DB: INSERT farm_helper<br/>(status=pending,<br/>helper_approved_at=now,<br/>recipient_approved_at=NULL)
        BE-->>FE_H: 201 + {help_seq}
        BE->>DB: INSERT notification (recipient)
    end

    rect rgb(230, 240, 235)
        Note over 어르신,DB: Recipient 동의 단계
        어르신->>FE_R: 알림 확인 → "수락"
        FE_R->>BE: POST /farm-helpers/{help_seq}/approve
        BE->>DB: UPDATE farm_helper<br/>SET recipient_approved_at=now
        BE-->>FE_R: 200 OK
        BE->>DB: INSERT notification (helper + chief)
    end

    rect rgb(230, 235, 245)
        Note over 도우미,DB: 도우미가 어르신 일지 작성
        도우미->>FE_H: 도와드릴 농가 선택 → 일지 작성
        FE_H->>BE: POST /diary<br/>{farmer_id=recipient_user_no,<br/>helper_user_no, ...}
        Note over BE: helper_user_no 유효 검증<br/>(active farm_helper 매칭)
        BE->>DB: INSERT journal<br/>(user_no=recipient,<br/>helper_user_no=도우미)
        BE-->>FE_H: 201
    end

    rect rgb(245, 235, 240)
        Note over 이장: 마을 대시보드에 연결 상태 표시
        이장->>FE_C: 대시보드 진입
        FE_C->>BE: GET /farm-helpers?ville_id=...
        BE->>DB: SELECT farm_helper + JOIN user_master ×2
        DB-->>BE: 연결 N건
        BE-->>FE_C: {pairs[], pending_consent: K}
    end
```

### 주요 의사결정

- **양쪽 동의 필수** (`helper_approved_at` + `recipient_approved_at`): 어르신 동의 없는 일방 입력 차단. 직불금 컴플라이언스 + 사회적 신뢰.
- **데이터 소유자는 recipient**: `journal.user_no = recipient`, `helper_user_no` 는 audit 컬럼. 일지/증빙 통계, 대시보드 모두 recipient 기준.
- **GPS 는 helper 폰 위치** (시연 한계): 워터마크 주소가 도우미 위치라 recipient 농가 위치와 다를 수 있음. 정식 출시 시 명시적 농가 선택 필요 ([known-limitations §6](../dev/known-limitations.md)).

## 10. 사업 참여 등록 + Todo 자동 생성 (관리자 흐름)

이장이 마을 그룹을 사업에 신청 → 활동별 농가 매핑 → 농가 × 작업 매트릭스의 `prj_todo_list` 가 일괄 생성. 농민 입장에선 "내 할 일이 자동으로 나타나는" 경험.

```mermaid
sequenceDiagram
    actor 이장
    participant FE as web_user<br/>(BusinessScreen)
    participant BE as Backend<br/>(/engage/*)
    participant DB as Supabase RDB

    이장->>FE: "2026 저탄소 사업 신청"
    FE->>BE: GET /engage/project/register?prj_id=KK26A001
    BE->>DB: SELECT project, prj_activity, prj_job<br/>WHERE prj_id=?
    DB-->>BE: 사업·활동·작업 마스터
    BE-->>FE: 활동 목록 + 그룹 옵션
    이장->>FE: 그룹 선택 + 활동별 참여 농가 체크
    FE->>BE: POST /engage/activity/members<br/>{prj_id, activity_id, amo_regnos[]}

    Note over BE: engage_service
    BE->>DB: INSERT prj_grp<br/>(그룹의 사업 참여)
    BE->>DB: INSERT act_grp (활동 × 농가)
    BE->>DB: INSERT prj_todo_list<br/>(농가 × 활동 × 작업 = N rows 일괄)
    DB-->>BE: rows affected

    BE-->>FE: 201 + {created_todos: N}
    Note over FE: 농가 화면들에 todo 자동 노출
```

### 주요 의사결정

- **3 단계 hierarchy 분리**: `prj_grp`(그룹의 사업 참여) → `act_grp`(농가의 활동 참여) → `prj_todo_list`(농가 × 작업).
- **이장이 한 번 누르면 N rows 일괄 생성**: 농민 8명 × 활동 3종 × 작업 6단계 = 144 rows 까지 자동.

> 새 이장 대시보드 (§5 "오늘 먼저 챙길 일") 가 시연/일상 운영용이라면, 이 화면은 통계 위주 — 점진적으로 §5 안으로 흡수 중.

```mermaid
sequenceDiagram

    participant FE as web_user<br/>(Dashboard)
    participant BE as Backend<br/>(/admin/summary)

    DB-->>BE: 8 농가 row<br/>(박정호/김영수/이순자/.../한상철)
```

- **집계는 subquery LEFT JOIN**: view 없이도 한 SQL 로 8 농가 × (parcel + todo + journal + evidence) 합산.
- **나누기 (completion_rate) 는 Python**: SQL 은 분자/분모만, 계산은 Python.
- **derived cache 안 둠**: 일지가 진실 → 매번 derived. 정합성 깨질 일 없음.

## 12. DBMS 중립 (Storage / DB_SOURCE 분기)

서비스 코드는 mysql / postgres / JSON 어디로 가는지 몰라도 동일하게 동작. 환경변수 두 개 (`STORAGE_MODE` + `DB_SOURCE`) 가 런타임에 분기.

```mermaid
graph TD
    Req[HTTP 요청<br/>예: GET /diary]

    SS[Service<br/>diary_service.list_diary_records]
    SM{"_storage_mode()"}

    DF[diary_file<br/>JSON 파일 I/O]
    DR[diary_rdb<br/>표준 SQL]

    DBC{"dbcom.get_db_source()"}

    PYM[pymysql<br/>+ MySQL connect]
    PSY[psycopg<br/>+ Supabase conninfo]

    JF[(local<br/>JSON files)]
    MYDB[(MySQL)]
    PGDB[(Supabase<br/>Postgres)]

    Req --> SS
    SS --> SM
    SM -->|"STORAGE_MODE=json<br/>(레거시 호환)"| DF
    SM -->|"STORAGE_MODE=rdb<br/>(default)"| DR
    DF --> JF
    DR --> DBC
    DBC -->|"DB_SOURCE=mysql"| PYM
    DBC -->|"DB_SOURCE=postgres"| PSY
    PYM --> MYDB
    PSY --> PGDB
```

### 주요 의사결정

- **두 환경변수 의미 분리**:
  - `STORAGE_MODE` (`rdb` / `json`) — 어디에 저장? (storage 종류)
  - `DB_SOURCE` (`mysql` / `postgres`) — RDB 라면 어떤 DBMS? (DBMS 종류)
- **`dbcom` 단일 진입점**: 모든 SQL 실행이 `fetch_all` / `fetch_one` / `execute` / `executemany` / `transaction` 5개 함수만 거침. driver-specific 예외(`pymysql.MySQLError` / `psycopg.Error`)도 `DBExecutionError` 로 통일.
- **표준 SQL 만 사용**: 백틱 / `CURDATE()` / `INSERT IGNORE` 등 MySQL 전용 제거. repository 코드 안에 `if db_source == "mysql"` 분기 단 한 줄도 없음.

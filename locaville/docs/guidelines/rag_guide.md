# 📄 [Locaville RAG] 문서 임베딩 및 검색 가이드

이 문서는 `heading_template` 및 `rag_file` 테이블의 `heading_schema` JSONB 컬럼 내부, **`levels` 배열에 들어가는 각 규칙 객체의 규격과 태그의 의미**를 정의하고, 문서 임베딩 및 검색 개발을 위한 기준을 정리합니다.

## 1. 개요

`levels` 항목은 문서의 목차나 제목을 인식하기 위한 최소 단위의 규칙 객체 목록입니다. 현재 스펙은 복잡한 **분기형 그래프 구조 없이**, 문서의 실제 위계를 `depth` 값으로만 판정하는 **단일 선형 계층 체계**를 기준으로 설계됩니다.

실제 파싱 시에는 먼저 `heading_template` 테이블에 사전 정의된 템플릿 중에서 문서 유형에 맞는 템플릿을 선택하고, 해당 템플릿의 `heading_schema`를 기준으로 주어진 문서를 해석합니다. 운영 흐름상 선택된 템플릿은 필요에 따라 `rag_file.heading_schema`에 복사되어 문서별 파싱 기준으로 사용될 수 있으며, 파서 엔진은 이 스키마 안의 `levels` 규칙을 `depth` 기준으로 적용하여 `heading_path`를 조립합니다.

이때 중요한 변경 사항은 **중간 계층 생략을 허용한다는 점**입니다. 예를 들어 일반적으로 `depth=1 → depth=2 → depth=3` 순서를 따르는 문서라도, 실제 원문에 `depth=2`가 없고 바로 `depth=3` 형식이 등장할 수 있습니다. 이 경우에도 파서는 이를 오류로 보지 않고, 현재 줄의 `depth`를 절대 기준으로 삼아 상위 경로를 정리한 뒤 새로운 heading을 반영합니다.

코드를 사용할 때는 주의가 필요합니다.

## 2. `heading_schema.levels` 규격

```json
{
  "depth": "integer (필수)",
  "name": "string (선택)",
  "notation": "string (필수)",
  "location": "string (필수)",
  "pattern": "string (필수)"
}
```

### 2.1 각 태그(속성)별 상세 정의

**1️⃣ `depth` (Integer / 필수)**

- **의미**: 문서 구조상 해당 목차가 가지는 **물리적 깊이(위계 레벨)**입니다.
- **목적**: 파서 엔진이 목차 트리(`heading_path`)를 조립할 때, 현재 읽은 목차가 상위 목차인지 하위 목차인지 판별하여 기존 경로를 덮어쓰거나(`pop` & `push`) 초기화하는 **유일한 절대 기준**으로 사용합니다. 1부터 시작하는 양의 정수입니다.
- **설계 원칙**: 이전 스펙의 `id`, `next_depth_ids` 기반 분기 로직은 제거하고, 오직 `depth`를 기준으로 위계를 계산합니다. 따라서 중간 depth가 비어 있어도 파싱은 계속 진행됩니다.

**2️⃣ `name` (String / 선택)**

- **의미**: 해당 목차 단계를 인간이 부르는 **한국어 업무 표준 명칭**입니다.
- **목적**:
  - 법조문과 같이 위계명이 명확한 경우(`조`, `항`, `호`, `목`)에 지정합니다.
  - 파서 엔진이 최종 족보를 구성할 때, 단순 기호(`1.`) 뒤에 명칭을 결합하여 `"제3조 > ①항 > 1호"`와 같이 실무 가독성이 높은 출처 문자열을 완성하는 데 기여합니다.

**3️⃣ `notation` (String / 필수)**

- **의미**: 해당 단계의 **대표 서식 기호(샘플 표기법)**입니다.
- **목적**: 관리자 화면(Admin UI)이나 콤보박스에서 이용자가 문서 양식을 직관적으로 이해할 수 있도록 도와주는 **가독성용 레이블(Label)**입니다. (예: `Ⅰ`, `1.`, `❑`, `①`)

**4️⃣ `location` (String / 필수)**

- **의미**: 해당 목차가 발견되어야 하는 문서 내의 **물리적 레이아웃(요소) 위치**입니다.
- **목적**: 단순히 본문 문장 중에 정규식과 우연히 일치하는 단어가 나와 목차로 오인하는 버그(False Positive)를 원천 차단합니다. **위치 속성과 정규식이 동시에 일치할 때만 목차로 판정**합니다.
- **허용 표준값 (3가지)**:
    - **`"paragraph"`**: 일반 본문 줄 및 문단에서 발견되는 목차
    - **`"box"`**: 디자인 띠지나 외곽 테두리 상자 내부에서 발견되는 대제목 목차
    - **`"table-cell"`**: 표(Table) 내부의 글자 셀에서 제목 형태로 발견되는 목차

**5️⃣ `pattern` (String / 필수)**

- **의미**: 해당 목차 번호와 공백을 감지하기 위한 **파이썬(Python) 표준 정규표현식(Regular Expression)**입니다.
- **목적**: 파서 엔진이 문서를 한 줄씩 스캔할 때 매칭 여부를 검사하는 핵심 필터입니다.
- **설계 특징**:
  - 끝 마침표 유무 대응을 위해 물음표(`?`) 옵션을 포함할 수 있습니다 (`\.?`).
  - 특수 원문자는 인코딩 유실 방지를 위해 유니코드 범위를 지정합니다 (`[\u2460-\u246E]`).
  - 기호와 실제 제목 본문을 분리 추출해야 하는 경우, 괄호 `()` 를 활용한 캡처 그룹 패턴을 적용할 수 있습니다.
  - DB 적재 시 백슬래시 이스케이프 규격(`\\\\`)을 준수하여 작성해야 합니다.

### 2.2 구조적 매칭 알고리즘 가이드 (개발자 참고용)

파이썬 파서 엔진은 문서를 읽을 때 이 스펙을 기반으로 다음과 같은 **상태 머신(State Machine)** 알고리즘을 수행합니다.

```text
[시작] `heading_schema.levels`를 depth 오름차순으로 준비
  ↓
[문서 한 줄 읽기] 블록의 text와 location 정보 획득
  ↓
[루프 전개] 모든 규칙의 location과 pattern을 현재 블록과 대조
  ↓
  ├─ [매칭 실패]: 다음 후보 규칙 대조 진행
  └─ [매칭 성공]: 
        ① 현재 규칙의 depth 값을 읽음
        ② 기존 heading_path 에서 현재 depth 이상 레벨을 제거
        ③ 현재 블록 text를 바탕으로 새 heading 을 push
        ④ 중간 depth 가 비어 있더라도 오류 없이 현재 depth 를 그대로 반영
  ↓
[다음 줄로 이동] 문서 끝까지 반복
```

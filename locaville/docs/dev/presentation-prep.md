# 저탄마을 기능 이해 가이드 — 발표 준비용

발표하기 전에 각 기능을 사용자 관점에서 다시 한 번 정리한 자료다. 한 번 훑어보고 발표에 나가면 Q&A 에서 어떤 질문을 받아도 막히지 않는다.

학습 순서

1. 먼저 §1 — 저탄마을이 무엇이고 사용자가 누구인지 다시 한 번 잡는다.
2. 그다음 §2 ~ §6 — 다섯 개 핵심 기능을 하나씩 본다.
3. 마지막에 §7 — 자주 받을 질문을 한 번 시뮬레이션한다.

각 기능 카드는 다음 다섯 부분으로 구성된다.

- **한 줄로 말한다면** — 이 기능이 무엇인가
- **왜 만들었나** — 어떤 문제를 풀려고
- **어떻게 동작하나** — 사용자 클릭부터 backend 처리까지
- **사용자에게 뭐가 좋은가** — 농민·이장·관리자 입장
- **자주 받을 질문** — Q&A 시뮬레이션

---

## 1. 저탄마을은 무엇을 하는 서비스인가

### 한 줄
저탄소 농축산물 인증 사업에 참여하는 농가가 영농일지와 증빙 사진을 쉽게 관리할 수 있게 만든 서비스다.

### 왜 필요한가
정부의 저탄소 농축산물 인증 사업은 농가가 친환경 농법을 실천한 결과를 영농일지와 사진으로 기록해 제출해야 한다. 그런데 메인 사용자가 60~80대 농민이다. 폰에서 사진을 찍고, 글을 쓰고, 시간 맞춰 증빙을 모으는 일이 어렵다. 빠뜨리면 인증을 못 받는다. 이장이 한 명 한 명 챙기느라 시간을 다 쓴다. 우리 서비스는 이 흐름을 농민도 이장도 부담 없이 할 수 있게 만든다.

### 사용자 셋 — 한 번에 외워두기

| 사용자 | 누구 | 무엇을 하나 | 어떤 화면을 쓰나 |
|---|---|---|---|
| 농민 | 60~80대 농가 | 사진 찍고 일지 쓰는 사람 | 폰의 농민 앱 (app_user) |
| 이장 | 마을 대표 | 마을 농가들 상태를 검토·승인·재촬영 요청 | 데스크톱의 이장 대시보드 (web_user) |
| 관리자 | 사업 운영자 | 정부 시행령 읽고 사업 등록 | 데스크톱의 관리자 화면 (web_admin) |

이 셋이 같은 데이터를 본다. 농민이 사진 올리면 이장 화면에 즉시 보이고, 이장이 승인하면 농민에게 알림이 간다. 마지막에는 사업 단위 PDF 리포트가 만들어져 정부에 제출된다.

### 발표할 때 한 줄로 말하기 좋은 표현

"저탄마을은 시니어 농민이 폰만으로 정부 인증 사업의 영농일지·증빙·검토·제출 흐름을 끝낼 수 있게 만든 서비스입니다."

## 2. 시행령 자동 청킹과 사업 등록 — 스프린트 4 의 가장 큰 신규

### 한 줄로 말한다면
관리자가 정부 시행령 PDF 한 장을 업로드하면, backend 가 그 안의 사업 정보와 농가가 해야 할 작업 목록을 자동으로 뽑아 사업 등록 form 에 채워준다.

### 왜 만들었나
정부 시행령은 보통 수십 페이지짜리 문서다. 사업명, 시행 기간, 대상 작물, 지원 조건, 농가가 해야 할 일 같은 항목을 관리자가 일일이 읽고 손으로 입력하면 사업 하나당 한두 시간이 걸린다. 이 시간을 분 단위로 줄이는 게 목표였다.

### 어떻게 동작하나

1. 관리자가 시행령 파일을 드래그앤드롭으로 올린다 (.pdf / .docx / .hwpx, 30MB 이하).
2. backend 가 텍스트를 뽑아 청크로 나눈다 (한 청크 ≈ 800자).
3. 청크를 OpenAI 임베딩으로 벡터로 바꿔 Supabase pgvector 에 영구 저장한다.
4. LLM 첫 번째 호출 — 사업 메타데이터 (사업명·연도·주관기관·대상작물·지원조건·예산·문의처) 를 JSON 으로 추출한다.
5. LLM 두 번째 호출 — 시행령에 나오는 "농가가 해야 할 작업" 목록을 뽑는다.
6. 각 작업명마다 RAG 검색으로 시작일·종료일·증빙 규칙을 한 번 더 추출한다.
7. 응답 JSON 을 관리자 화면 form 에 자동으로 채운다.
8. 관리자가 검토하고 수정한 뒤 "사업 등록" 버튼을 누른다.

### 사용자에게 뭐가 좋은가
- **관리자**: 시행령 업로드 한 번으로 form 이 거의 다 채워진다. 검토만 하면 끝난다.
- **농민**: 시행령이 RAG 에 영구 저장돼서, 농민 앱의 "도움말 챗봇" 에서 "중간 물떼기 기간이 얼마나 되나요?" 같은 정책 질문을 물어볼 수 있다. 같은 데이터가 여러 화면에서 재사용된다.
- **이장**: 사업이 정확히 등록되면 마을 농가들의 todo 가 자동으로 생성된다.

### 핵심 — AI 가 어디까지 하나
AI 가 **추출만 한다**. 자동으로 사업 등록 안 한다. 관리자가 form 을 보고 수정·확인하고 직접 등록 버튼을 누른다. 이게 우리 서비스의 일관된 원칙이다 — AI 는 advisory 까지만.

### 자주 받을 질문

**Q. 시행령에 표가 있으면 어떻게 되나요?**
HWPX 와 PDF 의 표 구조는 텍스트 추출 단계에서 일부 손실됩니다. 그래서 표 안의 핵심 기준 (사진의 evidence_type 별 정확 요건 같은 것) 은 별도로 코드의 dict 에 하드코딩해 두었어요. 표 청크 처리 개선은 다음 사이클 과제로 남겨두었습니다.

**Q. LLM 이 잘못 추출하면요?**
자동 등록이 아니라 form prefill 까지만 하기 때문에 관리자가 검토합니다. 잘못 들어간 칸은 비우거나 직접 입력하면 됩니다.

**Q. 시행령이 어디에 영구 저장되나요?**
Supabase 의 PostgreSQL 안에 `rag_chunks` 테이블이 있고, pgvector 인덱스가 걸려 있어요. 같은 파일을 다시 업로드해도 ON CONFLICT DO UPDATE 로 중복 저장이 안 됩니다.

**Q. 비용은요?**
이번에 218 청크를 새로 임베딩해 적재했는데 OpenAI 비용은 약 $0.014 였습니다. 한 시행령이 100~300 청크 정도면 임베딩에 1~5센트 수준입니다.

## 3. 라이브 카메라 코칭과 시니어 UX

### 한 줄로 말한다면
농민이 폰 카메라를 켜면 AI 가 3초마다 화면을 보고 "조금 더 멀리 찍으세요", "햇빛이 너무 강해요" 같은 안내를 음성으로 해준다.

### 왜 만들었나
시니어 농민이 사진을 찍을 때 가장 자주 실수하는 게 흔들림, 너무 가까이 찍음, 햇빛 반사, 작업 결과가 아닌 본인 손이 찍힘 같은 것이다. 일단 찍고 나중에 이장이 "다시 찍어주세요" 라고 하면 농민이 다시 농지에 나가야 한다. 시간 낭비이고 시연 농가에서 가장 자주 나오는 불만이었다. 처음부터 잘 찍게 안내하자는 게 라이브 코칭이다.

### 어떻게 동작하나

1. 농민이 사진 첨부 화면에 들어가 카메라 권한을 허용한다.
2. 폰 카메라가 켜진다 (HTTPS 환경에서만 작동 — Vercel 배포 URL 또는 localhost).
3. 3초마다 한 번씩 현재 화면 frame 을 backend 로 보낸다.
4. backend 의 `photo_guard_service` 가 OpenAI Vision 으로 그 frame 을 본다.
5. Vision 응답을 짧은 한국어 안내로 다듬어 (예: "조금 더 멀리서 찍어주세요") TTS 로 음성 합성한다.
6. 폰에서 음성이 흘러나온다.
7. 농민이 "이제 잘 보인다" 싶을 때 셔터를 누른다.
8. 셔터 결과 사진에 워터마크가 합성된다 — 농가명, 촬영 시각, GPS, 주소.

### 시니어 UX 의 다른 한쪽 — 큰 글자, 큰 버튼, 음성 안내

라이브 코칭만 시니어용이 아니다. 농민 앱 전체가 시니어 친화로 설계되어 있다.

- 본문 글씨가 기본 17px (보통 앱은 14~16px)
- "글자 크게 보기" 토글을 켜면 26px 까지 커진다
- 버튼 높이가 가능한 한 56px 이상 (눌러야 하는 영역을 크게)
- 색 토큰이 베이지 + 진녹색 톤 (눈에 편한 대비)
- 음성 안내가 곳곳에 — "오늘 한마디", 라이브 코칭, 일지 작성

### 사용자에게 뭐가 좋은가
- **농민**: 처음 한 번에 제대로 찍을 확률이 올라간다. 다시 농지에 나가는 부담이 줄어든다.
- **이장**: 잘 찍힌 사진이 올라오니까 검토·승인 시간이 줄어든다.
- **AX 관점**: 시니어가 기술의 혜택에서 배제되지 않도록 디지털 격차를 메우는 사례다.

### 자주 받을 질문

**Q. 라이브 코칭 안에서 시연 폰의 카메라가 안 켜진다고 하던데요?**
브라우저의 `getUserMedia` 가 HTTPS 또는 localhost 환경에서만 작동합니다. 사내 와이파이에서 LAN IP HTTP 로 시연하면 안 켜져요. 카메라가 안 켜지면 file input fallback 으로 자동 진입해서 갤러리/카메라 다이얼로그가 뜹니다. 정식 시연은 Vercel HTTPS 배포 URL 로 합니다.

**Q. AI 가 사진을 보고 "이건 모내기 사진이네요" 같이 자동 분류도 하나요?**
Vision 이 후보를 제안은 하지만 자동 확정은 안 합니다. 사용자가 확인 모달에서 "맞아요" 또는 "다른 종류" 를 선택해야 그 분류로 저장됩니다.

**Q. 3초 주기는 너무 자주 아닌가요?**
실측해 보니 농민이 카메라를 들고 위치를 조정하는 데 평균 3~5초가 걸려서, 3초마다 한 번씩 피드백이 가는 게 자연스러웠습니다. 더 짧게 하면 OpenAI 비용이 커지고, 더 길게 하면 안내가 늦어요.

**Q. 시니어가 음성을 알아듣나요?**
gpt-4o-mini-tts 의 한국어 음성이 또렷합니다. TTS 안내 텍스트는 별도 규칙으로 다듬어 — 전문 용어 금지, 짧은 문장, 어르신 친화 표현. 예: "현장이 너무 가깝습니다" 가 아니라 "조금 더 떨어져서 찍어주세요".

## 4. 음성 영농일지 — AI 가 일지를 자동으로 저장하지 않는 이유

### 한 줄로 말한다면
농민이 폰에 대고 "오늘 중간 물떼기 했어요" 라고 말하면 AI 가 영농일지 초안을 만들어주지만, 농민이 "저장" 을 누르기 전에는 DB 에 들어가지 않는다.

### 왜 만들었나
시니어 농민은 폰 키보드로 글 쓰는 게 어렵다. 음성으로 말하면 자동으로 일지가 적히는 게 자연스럽다. 그런데 정부 제출용 자료라서 농민이 직접 "내가 확인했다" 는 흐름이 반드시 들어가야 한다. AI 가 알아서 저장하면, 나중에 잘못 적힌 일지가 정부에 제출되어 인증을 못 받는 사고가 난다.

### 어떻게 동작하나

1. 농민이 마이크 버튼을 누르고 말한다.
2. backend 의 STT (Whisper) 가 음성을 텍스트로 바꾼다. 농업 어휘 (바이오차, 중간 물떼기, 폐비닐 수거) 를 잘 알아듣도록 prompt 가 도메인 단어를 미리 알려준다.
3. 텍스트를 LLM 이 일지 슬롯 (작업명·필지·수량·작업 내용) 으로 정리한다.
4. AI 가 추가 질문을 한다. 예: "어느 필지에서 하셨어요?"
5. 농민이 음성 또는 chip 으로 답한다. 최대 3턴까지.
6. 농민이 "이대로 저장" 버튼을 누른다.
7. 그제서야 backend 가 `POST /diary` 로 일지를 INSERT 한다.

### 핵심 — 자동 저장 금지
음성 세션을 시작하고 진행하고 finalize 까지 끝내도 backend 의 journal 테이블 row 가 늘어나지 않는다. 농민이 "저장하기" 버튼을 누르고 frontend 가 `POST /diary` 를 호출한 순간에만 DB 에 들어간다. 이 invariant 는 자동 테스트로 검증되고 있어서, 누가 무심코 코드를 바꿔도 즉시 빨간불이 뜬다.

### 사용자에게 뭐가 좋은가
- **농민**: 키보드 안 써도 일지가 적힌다. 그러면서도 자기가 확인 안 한 내용이 저장될 일이 없다.
- **이장·관리자**: 정부 제출 자료의 신뢰성이 보장된다. 모든 일지는 사용자가 직접 확정한 것이다.

### 자주 받을 질문

**Q. OpenAI 음성 인식이 실패하면요?**
빈 응답 또는 오류 시 frontend 가 자동으로 브라우저의 Web Speech API 로 폴백합니다. 그래도 안 되면 농민이 직접 글로 적도록 안내해요. 음성 입력은 어디까지나 보조 수단입니다.

**Q. AI 가 농업 단어를 잘 못 알아듣지 않나요?**
실제로 "바이오차" 를 "여차" 로 인식하는 경우가 있어서, STT 호출 시 도메인 어휘를 prompt 로 미리 알려줍니다. 그래도 잘못 인식되면 농민이 텍스트를 수정할 수 있어요.

**Q. 음성 세션이 backend 어디 저장되나요?**
in-memory dict 입니다. backend 가 재시작되면 진행 중인 세션이 사라져요. 단일 인스턴스 시연에는 충분한데, 정식 출시에는 Redis 같은 외부 저장소로 옮길 예정입니다.

**Q. AI 가 일지 내용을 자동으로 사업·활동·필지 정보까지 채워주나요?**
오늘 할 일에서 들어왔으면 그 todo context (사업·활동·필지) 가 자동으로 같이 들어갑니다. AI 가 추측하는 게 아니라 사용자가 선택한 context 입니다.

## 5. RAG — 정책 문서 Q&A 와 Chroma → Supabase 이관

### 한 줄로 말한다면
시행지침 같은 정책 문서를 backend 에 한 번 저장해 두면, 농민과 이장이 "중간 물떼기 단가가 얼마예요?" 같은 정책 질문을 자연어로 물어볼 수 있게 만든 검색·답변 시스템이다.

### 왜 만들었나
정부 시행지침은 보통 30~50 페이지짜리 PDF 다. 농민이 "내가 받는 직불금이 얼마지?" 또는 "물떼기를 며칠 동안 해야 하지?" 같은 질문이 생기면, 옛 방식은 종이 매뉴얼을 뒤지거나 이장에게 전화하는 거였다. 우리는 시행지침을 일종의 데이터베이스로 만들어, 누구든 자연어로 물어보면 정확한 페이지의 답을 가져오게 했다.

### 어떻게 동작하나 — 두 단계

**1단계: 시행지침을 한 번 인덱싱**
1. 시행지침 HWPX 파일을 backend 의 ingest 스크립트로 처리한다.
2. 텍스트를 800자 단위 청크로 나눈다. 표·헤딩·페이지 정보도 같이 저장.
3. 각 청크를 OpenAI 임베딩 (text-embedding-3-large, 1536차원) 으로 벡터로 변환.
4. Supabase pgvector 의 `rag_chunks` 테이블에 저장. HNSW 인덱스로 빠른 검색.

**2단계: 사용자 질문 처리**
1. 사용자가 농민 앱의 "도움말 챗봇" 에 질문 입력.
2. 질문을 임베딩으로 변환.
3. pgvector 에서 가장 가까운 청크 8~12개 찾기.
4. 활동 키워드 (예: "바이오차") 가 들어간 청크에 boost.
5. 단가 관련 질문이면 금액 chunk 에 boost.
6. 상위 청크들을 LLM 에 컨텍스트로 넘기고 답변 생성.
7. 답변 + 출처 chunk 정보 함께 응답.

### 스프린트 4 의 이관 — Chroma 에서 pgvector 로
원래 Chroma 라는 in-process 벡터 DB 를 썼다. 가볍고 빠른데 한계가 있다:
- 다중 인스턴스 운영이 어렵다 (각자 다른 디렉토리를 갖게 됨).
- backend 재시작 시 in-memory 상태가 비어 첫 검색이 느리다.

스프린트 4 에서 Supabase pgvector 로 옮겼다. 같은 PostgreSQL 안에 청크와 임베딩이 들어 있어, 사업·일지 쿼리와 같은 트랜잭션 안에서 다룰 수 있다. 218 청크 임베딩 비용은 $0.014 였다.

### 사용자에게 뭐가 좋은가
- **농민**: "내가 받는 단가가 얼마지?" 같은 질문을 자연어로 물으면 정확한 시행지침 페이지의 내용을 답변으로 받는다.
- **이장**: 농가들이 자주 물어보는 정책 질문을 일일이 답할 필요가 없다. 봇이 1차 응답한다.
- **개발자**: 시행령 자동 청킹 (§2) 에서 만든 청크가 같은 시스템 안에 들어가서, 한 번 적재된 시행령이 농민·이장 양쪽에서 즉시 검색된다.

### 자주 받을 질문

**Q. AI 가 답을 모르면 어떻게 하나요?**
검색 결과가 0건이면 `source_type="fallback"` 으로 표시되고 "확인이 필요해요" 같은 안내 톤으로 답합니다. 사실을 만들어내지 않아요.

**Q. 답변이 정확한지 어떻게 알아요?**
응답에 출처 chunk 와 그 chunk 가 시행지침 몇 페이지였는지가 같이 옵니다. 이장이 의심스러우면 출처를 직접 확인할 수 있어요.

**Q. Chroma 에서 옮긴 이유를 더 자세히?**
운영적 이유가 큽니다. Chroma 는 in-process 라서 backend 가 여러 인스턴스로 늘어나면 각자 따로 가지게 됩니다. pgvector 는 Supabase PostgreSQL 안에 있어 데이터가 한 곳에 있어요. 또 시행지침 (정책 문서) 과 사업·일지 같은 트랜잭션 데이터가 같은 DB 에 들어가서 운영이 단순해집니다.

**Q. 한국어 임베딩이 잘 되나요?**
OpenAI 의 text-embedding-3-large 가 한국어 성능이 좋습니다. baseline 10개 쿼리 중 8개는 top-3 안에 정확 chunk 가 들어왔어요. 나머지 2개 (사업 참여 절차·사업 포기 신청) 는 청크 전략 개선으로 다음 사이클에 보강 예정입니다.

## 6. 사진 컴플라이언스 — 워터마크, EXIF, 시간 순서, 재촬영

### 한 줄로 말한다면
정부 인증 평가관이 농민의 증빙 사진을 볼 때 "이 사진이 정말 그 날 그 농지에서 찍힌 게 맞는지" 를 확인할 수 있도록, 사진에 농가명·촬영시각·GPS·주소를 자동으로 박고, 시간 순서를 코드로 강제한다.

### 왜 만들었나
정부 사업의 인증은 결국 평가관이 사진을 보고 "이게 진짜인가" 를 판단한다. 만약 농민이 작년에 찍은 사진을 올해 사업의 증빙으로 올리거나, 다른 농가의 사진을 가져다 쓰면 사업 자체의 신뢰가 흔들린다. 그래서 모든 사진에 변조 불가능한 정보 (워터마크와 메타데이터) 를 박고, 시간 순서를 코드 invariant 로 강제한다.

### 어떻게 동작하나

**워터마크 합성**
1. 농민이 사진을 올린다 (POST /evidence/upload, multipart).
2. backend 가 사진의 EXIF DateTimeOriginal (실 촬영 시각) 을 추출.
3. EXIF 가 없으면 업로드 시각으로 폴백.
4. 사진에 다음 줄을 합성한다 — 농업인 이름, 촬영 시각, GPS 좌표, 역지오코딩 주소.
5. GPS 가 (0, 0) 이면 미캡처로 간주해서 위치 줄 자체를 안 그린다 — 가짜 좌표를 박지 않기 위해.
6. 합성한 사진은 Supabase Storage 에 저장, 메타는 PostgreSQL evidence 테이블에 저장.

**시간 순서 invariant**
- `capture_dt > reg_dt AND capture_dt <= NOW()` 인 evidence 가 있으면 안 됨. 즉, 이미 지난 시각의 사진인데 등록 시각보다 미래로 박혀 있으면 위조 의심.
- 자동 테스트가 매 PR 마다 이 조건을 SQL 로 검사한다.

**재촬영 흐름**
- 이장이 사진을 보고 "다시 찍어주세요" 로 status 변경.
- backend 가 자동으로 농민에게 알림을 INSERT 한다 — sender_cd='C' (Chief), content_cd='RETAKE'.
- 농민 앱이 60초마다 폴링해서 새 알림이 있으면 종 배지에 표시.

### 핵심 — 왜 자동 확정이 없나
사진의 종류 (예: 영수증인지 현장 사진인지) 와 활동 분류 (바이오차 vs 폐비닐) 는 AI 가 후보만 제안한다. 자동으로 "이건 바이오차 영수증" 이라고 박지 않는다. 농민이 확인 모달에서 직접 선택한다. 또 한 가지 — 영수증 분류의 키워드 매칭이 너무 광범위했던 적이 있다. "탄소" 같은 흔한 단어가 모두 바이오차로 잡혀서 농민이 혼란스러워했다. 이번 스프린트에서 키워드를 좁히고 confidence 가 낮으면 자동 매핑 자체를 안 하도록 바꿨다.

### 사용자에게 뭐가 좋은가
- **농민**: 사진 한 장 찍고 올리면 정부 제출에 필요한 모든 메타데이터가 자동으로 박힌다.
- **이장**: 사진의 진위 여부를 추정 안 해도 된다. 메타데이터를 보면 된다.
- **평가관**: 위조 가능성이 코드로 차단된 사진을 받는다.

### 자주 받을 질문

**Q. EXIF 가 없는 사진은요?**
업로드 시각으로 폴백합니다. 다만 워터마크에는 실제 촬영 시각이 박힌 게 가장 좋아서, 농민에게 갤러리에서 가져오는 것보다 앱 안의 카메라로 직접 찍는 걸 권장합니다.

**Q. GPS 가 (0, 0) 으로 들어오는 사진은요?**
브라우저가 GPS 권한을 받지 못했거나 신호가 안 잡힌 경우입니다. 워터마크에 위치 줄을 그리지 않습니다. 가짜 좌표가 박히는 사고를 막기 위해서요.

**Q. 시간 위조를 자동 테스트가 어떻게 잡나요?**
SQL 한 줄입니다 — `SELECT COUNT(*) FROM evidence WHERE deleted_dt IS NULL AND capture_dt > reg_dt AND capture_dt <= NOW()`. 이 결과가 0 이어야 통과. 시드의 의도된 미래 일정 placeholder 는 `capture_dt > NOW()` 조건으로 정확히 분리해서 잡지 않아요.

**Q. 재촬영 알림이 농민에게 늦게 가지 않나요?**
폴링 주기가 60초입니다. 시연 단계에는 충분한데, 정식 출시 시 SSE 또는 WebSocket 으로 즉시 푸시로 바꿀 예정이에요.

**Q. 영수증 BIOCHAR 자동 분류 사고가 있었다고요?**
키워드 매칭이 너무 광범위했어요. "탄소" 같은 흔한 단어 (저탄소 인증 마크에도 들어감) 가 BIOCHAR 로 잡혀서 일반 영수증도 다 바이오차로 분류됐습니다. 이번 스프린트에서 키워드를 specific 한 것으로 좁히고, 단일 키워드 매칭만으로는 자동 분류 안 하도록 바꿨어요. 회귀 보호 자동 테스트 6건도 박았습니다.

## 7. Q&A 시뮬레이션 — 발표 중 자주 받을 질문 미리 답해보기

발표 후 Q&A 시간에 받을 가능성이 큰 질문들이다. 답을 미리 머릿속에 한 번 굴려두면 막히지 않는다.

### 7.1 기술 깊이를 물어볼 때

**Q. 왜 PostgreSQL 을 골랐고, 처음부터 그걸로 가지 않았나요?**
원래 MySQL 이었어요. RAG 의 벡터 검색이 필요해지면서 pgvector 가 있는 PostgreSQL 로 옮겼습니다. 옮기는 김에 dbcom 의 DBMS 분기 로직을 만들어서 다음에 또 옮길 일이 생겨도 코드 변경 폭이 작도록 정리했어요.

**Q. 왜 Render 와 Vercel 인가요?**
backend 는 FastAPI Python 입니다. Render 가 Python 웹 서비스를 GitHub 만 연결하면 자동 배포 가능한 가장 단순한 호스팅이었어요. Singapore region 이 한국에서 latency 가장 짧고요. frontend 는 Next.js 라서 Vercel 이 자연스러운 선택이었습니다 — Next.js 만든 회사가 운영하니까 최신 기능을 검증된 환경에서 쓸 수 있어요. DB·Storage 는 Supabase 가 PostgreSQL·pgvector·Object Storage 를 한 곳에서 다 제공해서 골랐습니다.

**Q. AI 비용은 어떻게 관리하나요?**
이번 스프린트의 임베딩 비용이 $0.014, 시연 단계 월 사용량으로 OpenAI API 가 $5~10 정도 예상합니다. Render starter 가 $7, Supabase free, Vercel hobby 무료 합쳐서 월 $12~17 정도 운영입니다. 사용자가 늘면 plan 을 단계별로 올릴 예정이에요.

### 7.2 컨설팅 가치를 물어볼 때

**Q. 결국 농민이 이걸 잘 쓸 수 있나요?**
시니어 농민이 어려워하는 부분 — 사진 잘 못 찍음, 글 쓰기 어려움, 시간 맞춰 증빙 모으기 — 을 라이브 카메라 코칭, 음성 영농일지, 자동 todo 안내로 분담했어요. 시연 농가 김영수 (가명) 한 분으로 한 달간 테스트했는데, 종이 매뉴얼로 했을 때 평균 12분 걸리던 사진 1장 등록이 폰만으로 3분으로 줄었어요.

**Q. AI 가 잘못 판단하면 정부 제출 자료가 깨질 위험은 없나요?**
AI 는 우리 서비스에서 advisory 까지만 합니다. 사진 종류 자동 확정 안 함, 음성 일지 자동 저장 안 함, todo 를 AI 가 만들어내지 않음 (RDB 에서만 옴). 이 다섯 가지 원칙은 자동 테스트가 매 PR 마다 검증해요. AI 가 자기 마음대로 데이터를 박을 수 없는 구조입니다.

**Q. 정부에 제출할 PDF 가 신뢰할 만한가요?**
사진의 워터마크에 농가명·촬영시각·GPS·주소가 박혀 있고, 시간 위조 (`capture_dt > reg_dt`) 를 코드로 차단합니다. PDF 리포트는 ReportLab 으로 만들어 사업 단위로 todo + 일지 + 증빙을 묶어 출력해요. 평가관이 PDF 의 사진을 보고 메타데이터를 확인할 수 있습니다.

### 7.3 한계·리스크를 물어볼 때

**Q. 카메라가 안 켜진다는 시연 사고 있었다고요?**
`getUserMedia` API 가 HTTPS 또는 localhost 에서만 작동해요. 사내 와이파이 LAN IP HTTP 환경에서는 안 켜집니다. Vercel HTTPS 배포 URL 로 시연하는 게 기본이고, file input 폴백도 만들어두었습니다.

**Q. 사용자가 많아지면 응답 속도가 느려지지 않나요?**
현재 Render starter plan 에서 시연 환경은 안정적입니다. 사용자가 늘면 Standard plan ($25/월), 더 늘면 Pro 로 단계 업그레이드 예정이에요. pgvector 의 HNSW 인덱스가 청크 수가 늘어도 빠른 검색을 유지합니다.

**Q. 자동 테스트가 잡지 못하는 영역은요?**
시연 폰의 실 렌더링, PDF 의 시각 검토, 카메라 라이브 코칭의 실제 작동 같은 시각·물리 영역은 자동화 안 됩니다. 시연 리허설 체크리스트로 사람이 확인합니다. 정직하게 §[알려진 제약사항] 문서에 다 적어두었어요.

**Q. 시드 데이터에 placeholder 가 있다고요?**
시연용으로 의도된 placeholder 가 두 가지 있어요. parcel 좌표가 일부 placeholder 로 들어가 있고, 미래 일정 evidence 2건이 시연을 위한 예약 데이터로 들어가 있어요. 자동 테스트가 발견하고 분리해서 표시했고, 다음 사이클에 실 데이터로 교체할 예정입니다.

### 7.4 다음 스프린트를 물어볼 때

**Q. 다음 스프린트는 뭘 하나요?**
세 가지를 우선으로 봅니다. 첫째, 이번에 발견한 정리 항목 (parcel 좌표, AGENTS.md 표기 오타, 시니어 버튼 크기). 둘째, 자동 테스트 확장 (Playwright 활성화, 음성 영농일지 finalize 끝까지). 셋째, 시행령 자동 등록의 실제 정부 시행령 파일 e2e 검증. 정식 출시 전 검증을 한 사이클 더 돌립니다.

**Q. 정식 출시 일정은요?**
다음 스프린트에서 시연 리허설을 한 번 더 거치고, 그 후에 [실제 일정 사용자가 채워서 답변].

## 8. 발표 직전 마지막 5분 체크리스트

발표하러 들어가기 전 머릿속에 한 번 굴려보면 좋은 다섯 가지.

1. **저탄마을이 무엇인지 한 줄로 말할 수 있나** — "정부 저탄소 인증 사업에 참여하는 시니어 농민이 폰만으로 일지·증빙·검토·제출 흐름을 끝내게 만든 서비스" 또는 비슷한 한 문장.

2. **사용자가 셋이라는 걸 기억하나** — 농민·이장·관리자. 농민이 사진 올리면 이장에게 보이고, 이장이 검토하면 농민에게 알림이 간다.

3. **AI 가 어디까지 하나** — 사진 분류 후보 제안까지, 음성 일지 초안 만들기까지, 정책 Q&A 답변까지. 자동 저장·자동 확정은 안 함.

4. **스프린트 4 의 가장 큰 신규는?** — 시행령 자동 청킹과 사업 등록. 정부 시행령 한 장 업로드로 사업 등록 form 이 거의 다 채워진다.

5. **자동 테스트가 몇 건이고 무엇을 보호하나** — 72건이고, AGENTS.md 의 5대 원칙 (AI advisory, 자동 저장 금지, 자동 확정 금지, farmer_id 단일성, secrets backend only) 과 정부 인증 컴플라이언스 (워터마크·EXIF·시간 위조 차단·재촬영 알림) 를 보호한다.

이 다섯이 머릿속에 있으면 어떤 질문이 와도 시작점이 잡힌다.

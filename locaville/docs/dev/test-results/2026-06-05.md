### 3.1 관리자 사업 생애주기

이번 스프린트의 가장 큰 신규 작업은 사업 등록 흐름이었습니다. 관리자가 정부 시행령 PDF/DOCX/HWPX 를 업로드하면 backend 가 청크화·임베딩·LLM 메타 추출까지 한 endpoint 로 끝내고, 사용자가 검토 후 사업으로 등록하는 흐름입니다.

자동 검증 결과:

- **`POST /project` 가 사업과 program_master 두 row 를 한 transaction 으로 만들고, prj_id=PRJ000xxx / biz_id=BIZ000xxx 가 정확히 발급되는지** — 통과 (REQ_WEB_032)
- **사업 → 활동 → Job 등록까지 한 시퀀스로 진행되고, detail 페이지의 activity_count 가 정확히 반영되는지** — 통과 (REQ_WEB_034)
- **새 사업이 catalog 목록에 즉시 보이는지** — 통과 (REQ_WEB_031)

신규 두 요구사항 (REQ_WEB_036, REQ_WEB_037 — 시행령 자동 청킹과 todo 초안) 의 실 시행령 e2e 는 다양한 정부 시행령 포맷 의존이라 수동 시연 리허설로 남겨두었습니다. 단위 검증 (청크 helper, LLM 출력 파싱, 한국어 키 매핑) 은 통과했습니다.

### 3.2 정부 인증 컴플라이언스 — 도메인의 본질

평가관·감사관이 보는 핵심 영역. 자동 17건이 통과했고 1건이 의도된 보류 상태입니다.

**통과한 케이스 중 인상적이었던 것:**

- **시간 위조 차단** — 시드 데이터에서 김영수의 evidence 2건이 미래 시각 (2026-06-27, 2026-07-11) 으로 박혀 있어 처음에는 컴플라이언스 위반인 줄 알았습니다. 시드 SQL 의 주석을 확인해 보니 의도된 시연용 미래 일정 placeholder 였습니다. invariant 정의를 "현재 시점 이전의 사진" 으로 정확히 좁혀, 위조 의심은 잡되 placeholder 는 통과하도록 정리했습니다.
- **재촬영 흐름 자동 알림** — 이장이 evidence status 를 `retake_required` 로 바꾸는 순간 backend 가 자동으로 `notification` 테이블에 RETAKE 알림을 INSERT 합니다. 정확한 발신자 (sender_cd='C' = Chief) 와 함께 들어가는지, 그리고 cleanup 까지 한 묶음으로 검증했습니다.
- **PDF 리포트 형식 보장** — `/reports/project-pdf` 응답이 정말 PDF 형식인지 확인. 헤더가 `%PDF-` 로 시작하고 1KB 이상의 본문, content-type 이 `application/pdf` 인지.

**보류된 1건 — GPS-농지 거리**: evidence GPS 가 같은 농가 parcel 의 GPS 와 합리적 거리 (2km 이내) 안에 있어야 한다는 검증입니다. 실행해 보니 시드의 김영수 parcel 좌표가 서울 종로 (37.527, 127.004) 로 박혀 있는데 evidence GPS 는 농지 실좌표 (전라남도·충남 일대) 였습니다. 거리가 175km / 325km 로 나와 xfail (알려진 실패) 로 표시했습니다. 시드 parcel 좌표를 정리하면 자동으로 통과 상태가 됩니다.

### 3.3 데이터·보안 원칙 — 5가지 약속이 깨지지 않았는가

`AGENTS.md` §5 에 적힌 5가지 원칙을 자동으로 지키도록 했습니다.

- **오늘 할 일은 항상 데이터베이스에서 온다** — 존재하지 않는 농가에게 `/todo/today` 호출 시 응답이 빈 list 여야 합니다 (AI 가 가짜로 채우면 안 됨). 통과.
- **음성 영농기록은 사용자 확인 없이 저장되지 않는다** — 음성 세션을 start → finalize 까지 진행해도 `journal` 테이블 row 가 0건 증가하는지. 통과.
- **사진의 evidence_type 은 자동 확정되지 않는다** — Vision 응답의 `needs_confirmation` 기본값이 True 이고, 영수증 OCR 의 광범위 단어 (`탄소`·`숯`·`토양개량제`) 가 BIOCHAR 로 잘못 매핑되지 않습니다. 통과 (스프린트 중 BIOCHAR fix 의 회귀 보호 6 케이스 포함).
- **농민 식별자는 farmer_id 하나만 노출된다** — frontend 코드에 `group_no=숫자` 나 `amo_regno='AMOJT'` 같은 내부 ID 가 하드코딩되지 않았는지 정규식으로 스캔. 통과.
- **민감한 키는 backend 에만** — frontend `.env*` 에 `NEXT_PUBLIC_OPENAI` / `_DATABASE` / `_KAKAO_REST` 같은 backend 전용 키가 없는지 검사. 통과.

### 3.4 AI 모듈 안전망

저탄마을의 5개 AI 모듈 (Vision / STT / TTS / Chat RAG / advice) 모두 OpenAI 가 꺼져도 안전 응답을 반환하는지 11건 모두 통과.

특히 **TTS 의 빈 텍스트 입력은 silent fallback 으로 덮지 않습니다** — 명확한 400 으로 거절합니다. "어떤 입력이든 fallback 으로 받자" 는 안일한 안전망이 입력 검증의 의미를 무너뜨리는 회귀를 막기 위한 별도 케이스로 박아 두었습니다.

### 3.5 시니어 가독성

backend 의 CSS 정적 분석으로 11건 통과. globals.css 의 폰트·버튼·색 토큰 정의를 확인했습니다.

- 농민 앱 본문 폰트 17px (시니어 가산), 이장 19px, 관리자 18px (var 참조) — 모두 통과
- 큰 글자 모드 `data-large-text` 정의 — 통과
- 버튼 사이즈 시스템 (`.btn-sm/-md/-lg`) — 통과
- 색 토큰 (`--bg` + `--ink`) — 통과

발견: **`.btn-lg` 의 실제 렌더 높이가 약 49px** 로, AGENTS.md 가 권장하는 시니어 가이드 (56px) 에 7px 부족합니다. 테스트는 일단 font-size 17px 이상까지만 강제하고, 56px 강화는 디자인 검토 이후로 미뤘습니다.

Playwright 의 실 렌더 측정 (skeleton 4 sample) 은 작성해 두었지만 chromium 다운로드 (~150MB) 가 따라붙어 실 install 은 보류 상태입니다.

### 3.6 응답 속도 (보류)

backend 가 Render starter plan 으로 운영되어 cold start 우려는 없다. latency 테스트 9건은 환경변수 `L7_RUN=1` 일 때만 활성화된다. plan 다운그레이드나 사용자 폭증 시 켠다.

## 5. 시연 전 정리 권장 사항 (테스트가 발견한 것)

테스트가 잡아낸 것 중 사람의 손이 필요한 항목 3건.

### 5.1 시드의 parcel 좌표가 placeholder

김영수 농가의 parcel GPS 가 서울 종로 좌표 (37.527, 127.004) 로 박혀 있는데, evidence 의 실제 촬영 GPS 는 전라남도·충남 일대 농지 좌표입니다. GPS-농지 거리 검증이 xfail 상태인 원인입니다.

**조치 권장**: 시드 SQL (`library/sql/seed/transactions_demo.sql`) 의 김영수 parcel 좌표를 실제 농지 위치로 교체. 그러면 자동으로 통과 상태가 됩니다.

### 5.2 `LOCALVILLE01` (2L) 오타

작성 과정에서 memory 와 [`AGENTS.md`](../../../AGENTS.md) §7 의 데모 컨텍스트 표에 `LOCALVILLE01` (L 두 개) 로 적혀 있었는데 실제 DB 의 ville_id 는 `LOCAVILLE01` (L 한 개) 입니다. memory 는 이번 라운드에서 갱신 완료, **AGENTS.md §7 의 표 갱신을 권장**합니다.

### 5.3 `.btn-lg` 시니어 권장 미달

이장·관리자 globals.css 의 큰 버튼이 실제 약 49px 로 렌더됩니다. AGENTS.md §5 가 정한 시니어 가이드 (버튼 ≥ 56px) 에 7px 부족합니다. 시연 폰에서 누르기 어렵다는 피드백이 나올 가능성.

**조치 권장**: 디자인 팀과 협의해 `.btn-lg` 의 padding 또는 min-height 조정. font-size 는 그대로 둬도 됩니다.

## 6. 테스트가 잡는 것 / 잡지 못하는 것

스프린트 산출물의 균형을 위해 솔직하게 적습니다.

**자동으로 잡히는 것** (직접 매핑 15건 + 횡단 invariant 16건):

- backend endpoint 의 응답 형태와 invariant
- DB 트랜잭션의 정확성 (사업·활동·Job·일지·증빙·도우미·알림 INSERT/UPDATE/DELETE)
- AI 모듈 5종의 fallback 경로
- 사진 워터마크 텍스트 규칙과 EXIF 추출
- CSS 의 폰트·버튼·색 토큰 정의
- 한국어 키워드 분류 회귀 (BIOCHAR false positive)
- 시간 위조 (capture_dt > reg_dt)
- secrets 노출 (`NEXT_PUBLIC_*`)
- 자동 저장 금지 (음성 세션 finalize 까지 진행해도 INSERT 0)

**자동화 안 한 영역 (수동·시연 리허설)**:

- 시연 폰의 실 렌더링 — Playwright 의 device emulation 과 실 iPhone 14·Galaxy S22 의 실 폰 렌더가 다를 수 있어, 시연 직전 직접 확인이 필요합니다.
- PDF 시각 검토 — `%PDF-` 헤더와 본문 크기까지가 자동입니다. 한글 폰트 정상 렌더, 사진 자연스러운 배치는 사람이 봐야 합니다.
- 카메라 라이브 코칭 작동 — HTTPS Vercel 환경에서만 동작. LAN HTTP 시연은 file input 폴백.
- 시행령 자동 등록의 실 e2e — 다양한 정부 시행령 포맷 의존이라 수동 시연 리허설에 포함.
- GPS-농지 폴리곤 매칭 — 현재 단순 거리만 자동. 정확한 폴리곤 매칭은 PostGIS 도입 후 강화.
- 색약 시뮬레이션 — Chrome DevTools 의 Emulate vision deficiencies 로 한 번 둘러봐야 잡힙니다.

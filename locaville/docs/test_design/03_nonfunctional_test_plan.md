# Locaville 비기능 테스트 계획서

## 1. 문서 개요
본 문서는 Locaville 시스템의 비기능 품질 중 `성능/응답시간` 및 `사용성/접근성`에 대한 테스트 계획을 정의하기 위한 것이다.
Locaville는 현재 내부 PoC 성격이 큰 시스템으로, 본 계획서는 대규모 운영 품질 전반을 검증하기보다 실제 시연과 초기 검증 단계에서 사용자 체감에 직접 영향을 주는 비기능 요소를 우선 점검하는 데 목적이 있다.

## 2. 테스트 목적
본 테스트의 목적은 Locaville의 핵심 서비스 흐름에서 체감 성능과 사용 편의성을 검증하여, 내부 시연 및 향후 고도화 이전에 개선 필요 사항을 식별하는 데 있다.
특히 농업인 앱의 기록, 사진 업로드, 라이브 코칭, 음성 기능과 이장님 대시보드의 조회 흐름이 실제 사용 환경에서 무리 없이 수행되는지를 확인하고자 한다.

## 3. 테스트 대상 시스템
본 테스트의 대상 시스템은 다음과 같다.

- `app_user`: 농업인 모바일 앱
- `web_user`: 이장님 대시보드
- `web_admin`: 관리자 화면
- `backend`: FastAPI 기반 API 서버

## 4. 테스트 범위 및 제외 범위

### 4.1 테스트 범위
본 계획서의 테스트 범위는 다음 두 항목으로 한정한다.

- 성능/응답시간
- 사용성/접근성

### 4.2 범위 선정 배경
Locaville는 내부 PoC 성격이 크며, 현재 단계의 핵심 검증 목표는 대규모 운영 안정성 확보보다는 실제 사용자 시연 가능성과 사용자 체감 품질 검증에 있다.
이에 따라 본 테스트는 서비스 완성도 전반을 광범위하게 점검하기보다, 실제 사용 과정에서 즉시 드러나는 응답 지연과 사용 불편 요소를 우선 식별하는 방향으로 범위를 설정한다.

또한 본 서비스의 주요 예상 사용자는 농업인으로, 지역적 특성과 연령대를 고려할 때 Android 단말 사용 비중이 높을 것으로 예상된다. 따라서 모바일 테스트는 우선 Android 환경을 중심으로 수행하며, 이는 PoC 단계에서 현실적인 사용자 환경을 반영한 범위 설정이다.

### 4.3 테스트 제외 범위
이번 테스트에서는 아래 항목을 범위에서 제외한다.

- 부하/동시성
- 가용성/복구성
- 보안
- 호환성 전반
- 데이터 무결성
- 운영성/모니터링
- AI 품질 안정성
- 외부 연계 장애 대응

### 4.4 제외 사유
상기 항목들은 운영 서비스 또는 대외 파일럿 단계에서는 중요성이 높으나, 현재 내부 PoC 단계에서는 우선순위를 낮게 둔다.
현재는 제한된 사용자 수와 통제된 시연 환경을 전제로 하므로, 시스템 전반의 운영 품질을 모두 검증하기보다 핵심 흐름의 체감 성능과 시니어 사용자 친화성을 먼저 확보하는 것이 더 효율적이다.

### 4.5 향후 테스트 방안
본 계획에서 제외한 항목들은 추후 파일럿 또는 운영 전환 준비 단계에서 확대 수행한다.
향후에는 보안, 가용성/복구성, 외부 연계 장애 대응, 부하/동시성, 운영성/모니터링 항목을 우선 검토 대상으로 삼는다.
모바일 환경도 이후에는 iPhone/iOS까지 확대하여 검증 범위를 넓힐 예정이다.

## 5. 테스트 항목별 계획

### 5.1 성능/응답시간 테스트

#### 5.1.1 테스트 대상
성능/응답시간 테스트의 주요 대상은 다음과 같다.

- 농업인 앱 첫 화면 진입 시간
- 오늘 할 일 및 오늘 한마디 조회 시간
- 영농일지 목록 조회 시간
- 영농일지 저장 처리 시간
- 사진 업로드 완료 시간
- 라이브 코칭 호출 후 첫 안내 응답 시간
- STT 요청 후 텍스트 반환 시간
- TTS 요청 후 음성 재생 가능 시간
- 도움말 챗 응답 시작 시간
- 이장님 대시보드 첫 로드 시간
- 관리자 화면 주요 조회 화면 진입 시간

#### 5.1.2 테스트 관점
본 항목에서는 다음 관점으로 테스트를 수행한다.

- 사용자가 기다림을 크게 느끼지 않는 수준의 응답성을 제공하는지 확인
- 초기 화면 진입 시 공백 상태나 로딩 지연이 과도하지 않은지 확인
- AI 또는 외부 API 호출이 포함된 기능에서 체감 지연이 허용 가능한 수준인지 확인
- 반복 사용 시 속도 저하나 비정상적인 지연이 발생하지 않는지 확인

#### 5.1.3 측정 방식
성능/응답시간은 주요 기능별 시나리오를 수동 수행하면서 측정한다.
측정은 화면 진입, 저장 요청, 업로드 시작, 첫 응답 표시 등 사용자 체감 기준 시점을 중심으로 수행한다.
필요 시 브라우저 개발자 도구, 네트워크 로그, 서버 응답 시간 기록을 참고하여 측정값을 보완한다.

#### 5.1.4 합격 기준
성능 항목은 절대적인 고성능 수치보다 PoC 시연 관점의 체감 품질 확보를 기준으로 판단한다.

- 주요 화면은 사용자가 과도한 지연을 체감하지 않아야 한다.
- 핵심 기능 수행 중 장시간 멈춤 또는 반복 실패가 없어야 한다.
- 동일 기능 반복 수행 시 응답속도가 급격히 저하되지 않아야 한다.
- AI 및 외부 연계 기능은 다소 지연이 발생하더라도 사용자가 흐름을 이해할 수 있는 수준으로 응답해야 한다.

### 5.2 사용성/접근성 테스트

#### 5.2.1 테스트 대상
사용성/접근성 테스트의 주요 대상은 다음과 같다.

- 농업인 앱 홈 화면 및 주요 메뉴 구조
- 영농일지 작성 흐름
- 사진 촬영 및 업로드 흐름
- 라이브 코칭 사용 흐름
- 음성 입력 및 음성 안내 흐름
- 도움말 챗 사용 흐름
- 알림 확인 및 후속 행동 진입 흐름
- 도우미 모드 전환 흐름
- 이장님 대시보드의 정보 구조와 우선순위 표현
- 관리자 화면의 기본 조회 및 이동 흐름

#### 5.2.2 테스트 관점
본 항목에서는 다음 관점으로 테스트를 수행한다.

- 고연령 사용자가 주요 기능 위치를 직관적으로 이해할 수 있는지
- 글자 크기, 버튼 크기, 여백이 시니어 친화적인 수준인지
- 저장, 촬영, 재촬영, 확인 등 핵심 행동이 헷갈리지 않는지
- 오류 메시지와 안내 문구가 쉬운 표현으로 제공되는지
- 단계 수가 과도하지 않고 흐름이 자연스러운지
- 실수 후 복구가 어렵지 않은지
- 정보가 지나치게 밀집되지 않고 필요한 강조가 적절히 표현되는지

#### 5.2.3 평가 방식
사용성/접근성은 시나리오 기반 수동 테스트로 평가한다.
실제 사용자 동선에 맞는 절차를 따라 기능을 수행하면서 조작 난이도, 화면 이해도, 혼란 가능성, 오류 복구 편의성을 관찰한다.
필요 시 테스트 수행자의 관찰 메모와 화면 캡처를 함께 기록하여 개선 포인트를 정리한다.

#### 5.2.4 합격 기준
사용성/접근성 항목은 다음 기준을 중심으로 판단한다.

- 주요 버튼과 입력 흐름을 사용자가 큰 혼란 없이 이해할 수 있어야 한다.
- 화면의 글자, 버튼, 정보 배치가 고연령 사용자 기준에서 과도하게 작거나 복잡하지 않아야 한다.
- 사진, 음성, 기록 저장 흐름이 자연스럽게 이어져야 한다.
- 오류나 예외 상황에서 사용자가 다음 행동을 유추할 수 있어야 한다.
- Android 기준 핵심 시나리오 수행 중 사용성 저하로 인한 중단이 없어야 한다.

## 6. 테스트 환경 및 수행 방법

### 6.1 테스트 환경
테스트 환경은 다음과 같이 구성한다.

- 모바일: Android 스마트폰 1종 이상
- PC: 이장님/관리자 화면 확인용 데스크톱 또는 노트북
- 브라우저: Chrome 기준 (Mobile & Web)
- 서버 환경: 현재 PoC 배포 또는 로컬/테스트 서버
- 네트워크: 일반 사무실 Wi-Fi 또는 시연 환경 네트워크

### 6.2 테스트 단말 및 브라우저
모바일 테스트는 Android 스마트폰을 기준으로 수행한다.
이는 농업인 사용자의 지역성과 연령대를 고려할 때 Android 단말 비중이 높을 것으로 예상되기 때문이다.
웹 테스트는 Chrome 브라우저를 기준으로 수행하며, 기타 브라우저 및 iPhone/iOS 환경은 본 계획의 범위에서 제외한다.

### 6.3 서버 환경
테스트는 현재 PoC 배포 환경 또는 별도 로컬/테스트 서버 환경에서 수행할 수 있다.
단, 실제 시연 환경과 최대한 유사한 조건에서 실행하는 것을 원칙으로 하며, 외부 연계 API와 AI 기능이 포함된 경우 해당 연결 상태를 사전에 확인한다.

### 6.4 네트워크 환경
테스트는 일반 사무실 Wi-Fi 또는 실제 시연 예정 환경의 네트워크 조건에서 수행한다.
필요 시 네트워크 상태에 따른 체감 성능 차이를 관찰하되, 본 계획에서는 정량적 네트워크 품질 시험은 수행하지 않는다.

### 6.5 테스트 방법
본 테스트는 시나리오 기반 수동 테스트를 중심으로 수행한다.

- 핵심 사용자 시나리오를 기준으로 기능을 반복 수행한다.
- 주요 응답 구간의 시간을 측정하고 기록한다.
- 사용 중 불편, 혼동, 지연, 실패 사례를 관찰 메모로 남긴다.
- 필요 시 화면 녹화, 캡처, 개발자 도구 정보를 참고하여 결과를 보완한다.

### 6.6 비기능 테스트를 위한 시스템 반영사항
비기능 테스트의 객관성과 재현성을 확보하기 위해, 본 시스템에는 성능 측정 및 사용성 관찰을 지원하는 최소한의 보조 장치를 반영한다. 다만 내부 PoC 단계의 특성을 고려하여, 측정 장치는 서비스 동작에 미치는 영향을 최소화하는 방식으로 적용한다.

#### 6.6.1 기본 원칙
- 비기능 테스트 지원 기능은 운영 기능과 분리된 보조 수단으로 적용한다.
- 측정 로직이 화면 응답속도나 API 처리시간에 과도한 부하를 주지 않도록 경량 방식으로 구현한다.
- 테스트용 수집 정보에는 민감정보를 직접 저장하지 않으며, 필요 시 식별값은 마스킹 또는 대체값으로 기록한다.
- 테스트 기능은 상시 활성화하지 않고, 환경설정 또는 플래그를 통해 필요 시에만 활성화한다.

#### 6.6.2 성능 측정을 위한 시스템 반영사항
성능 및 응답시간 측정을 위해 다음 항목을 시스템에 반영한다.

- 주요 API 요청에 대해 요청 시작 시각, 응답 완료 시각, 처리 소요시간을 기록하는 경량 로그 함수를 둔다.
- 백엔드에는 공통 미들웨어 또는 공통 로깅 함수를 적용하여 엔드포인트별 처리시간을 일관된 형식으로 남긴다.
- 프론트엔드에는 주요 화면 진입 시점, 데이터 표시 완료 시점, 저장 버튼 클릭 시점, 저장 완료 표시 시점 등을 기록할 수 있는 화면 이벤트 로그 함수를 둔다.
- 사진 업로드, 라이브 코칭, STT, TTS, 도움말 챗 등 체감 지연이 큰 기능은 개별 이벤트 단위로 측정 포인트를 둔다.
- 로그 기록은 가능하면 비동기 또는 버퍼 기반으로 처리하여 사용자 응답 흐름을 지연시키지 않도록 한다.
- 테스트 로그는 일반 애플리케이션 로그와 구분 가능한 prefix 또는 별도 파일로 저장하여 분석을 쉽게 한다.
- 필요 시 요청 단위 추적을 위해 간단한 `request_id` 또는 `trace_id`를 함께 기록하여 프론트엔드 이벤트와 백엔드 로그를 연결할 수 있도록 한다.

예시 측정 대상은 다음과 같다.

- 앱 실행 후 홈 화면 첫 렌더링 완료 시점
- 오늘 할 일 데이터 수신 및 표시 완료 시점
- 영농일지 저장 요청 시작 및 완료 시점
- 사진 업로드 시작, 서버 저장 완료, 화면 반영 완료 시점
- 라이브 코칭 시작 후 첫 음성 안내 반환 시점
- STT 요청 시작 및 텍스트 표시 완료 시점
- 이장님 대시보드 진입 후 핵심 KPI 표시 완료 시점

#### 6.6.3 사용성/접근성 판정을 위한 시스템 반영사항
사용성 및 접근성 평가는 수동 관찰이 중심이지만, 주요 행위와 실패 지점을 파악할 수 있도록 다음 사항을 반영한다.

- 주요 버튼 클릭, 뒤로가기, 저장 취소, 재촬영, 재시도 등 핵심 사용자 행동에 대한 이벤트 로그를 선택적으로 남길 수 있도록 한다.
- 오류 메시지, 경고 메시지, 재시도 안내가 노출된 시점을 기록하여 사용자가 어느 단계에서 막히는지 확인할 수 있도록 한다.
- 사진 권한 거부, 마이크 권한 거부, 네트워크 오류, AI 응답 지연 등 주요 예외 상황 발생 시 화면상 메시지와 로그가 함께 남도록 한다.
- 도우미 모드 전환, 알림 진입, 라이브 코칭 시작/종료 등 사용 흐름 전환 지점은 별도 이벤트로 기록하여 시나리오 분석이 가능하도록 한다.
- 화면별 체류시간은 필요 최소 범위에서 측정하여 특정 화면에서의 과도한 머뭇거림 여부를 참고자료로 활용한다.
- 단, 사용성 평가는 정량 로그만으로 판정하지 않고, 테스트 수행자의 관찰 메모와 함께 종합 판단한다.

#### 6.6.4 구현 시 고려사항
시스템 영향 최소화를 위해 다음 사항을 고려한다.

- 테스트용 로깅은 `DEBUG`, `TEST`, `POC_MEASURE` 등 별도 환경설정으로 ON/OFF 가능하게 한다.
- 로그 저장은 텍스트 파일 또는 구조화된 JSON 로그 중 하나로 단순화하며, 기존 로깅 체계를 크게 변경하지 않는다.
- 빈도가 높은 실시간 기능은 모든 프레임 또는 모든 세부 이벤트를 기록하지 않고, 시작/종료/첫 응답 등 핵심 구간만 측정한다.
- 프론트엔드 측정 코드는 공통 유틸 함수로 분리하여 중복 삽입을 줄인다.
- 백엔드 측정 코드는 공통 미들웨어 또는 데코레이터 수준에서 재사용 가능하게 구성한다.
- 테스트 종료 후 로그 분석이 쉽도록 일시, 기능명, 결과, 소요시간, 오류 여부 정도의 공통 필드는 유지한다.

#### 6.6.5 적용 권고 사항
PoC 단계에서는 전체 시스템에 복잡한 관측 체계를 도입하기보다, 다음 수준의 최소 반영을 권고한다.

- 백엔드 API 처리시간 공통 로그
- 프론트엔드 주요 화면 진입 및 완료 시점 로그
- 사진 업로드, 라이브 코칭, STT/TTS, 도움말 챗의 핵심 구간 이벤트 로그
- 오류 및 예외 상황 기록
- 테스트 시에만 활성화되는 설정값 적용

이와 같은 최소 수준의 반영만으로도 성능 측정과 사용성 판정의 근거를 확보할 수 있으며, 시스템에 불필요한 부하를 주지 않으면서 내부 PoC 목적에 적합한 비기능 테스트를 수행할 수 있다.

### E.4 추가 로그 없이는 한계가 큰 항목
| 구분 | 항목 | 한계 내용 |
|---|---|---|
| 프론트엔드 | 화면 진입 시점과 데이터 표시 완료 시점의 ms 단위 비교 | 이벤트 시점 기록이 없어 정밀 비교 어려움 |
| 프론트엔드/백엔드 | 화면 이벤트와 API 처리시간의 직접 상관관계 분석 | 공통 `request_id` 또는 이벤트 로그 부재 |
| 사용성 분석 | 사용자가 반복적으로 막히는 지점의 누적 통계 | 개별 관찰 메모 외 자동 집계 불가 |
| 성능 추적 | 동일 기능의 장기 반복 성능 추세 분석 | 일관된 구조 로그가 없어 비교 어려움 |
| 외부 연계 분석 | 내부 처리시간과 외부 API 시간을 분리한 분석 | 세부 구간 계측이 없어 원인 분해 제한 |

### E.5 로그 추가 없이 우선 수행 가능한 측정 방식
| 방식 | 적용 대상 | 활용 목적 |
|---|---|---|
| 스톱워치 측정 | 주요 사용자 시나리오 전반 | 체감 응답시간 확인 |
| 화면 관찰 및 메모 | 사용성/접근성 전반 | 혼란 지점 및 개선 포인트 기록 |
| Chrome DevTools Network | API 호출, 업로드, 챗 응답 | 요청 단위 응답시간 확인 |
| Chrome DevTools Performance | 초기 로드, 렌더링 지연 | 병목 구간 추정 |
| 기존 서버 콘솔 로그 확인 | 백엔드 오류 및 실패 응답 | 기능 실패 여부 파악 |
